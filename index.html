Projeto Técnico II

Sistema de Horas Complementares (SHC)

Desenvolvimento de uma solução web para centralização e gestão de atividades complementares no ensino superior.

Autores: Leonardo de Oliveira e Maria Eduarda Schmidt

Orientadoras: Profª Msc. Simone Regina da Silva e Profª Daniela Amorim

Faculdade Municipal de Palhoça (FMP)

Análise e Desenvolvimento de Sistemas

O Problema e o Desafio Institucional

01

Processo Manual

Uso de formulários físicos e entrega presencial de documentos, gerando manipulação sucessiva dos mesmos dados em diferentes etapas.

02

Descentralização

Necessidade de transcrição manual de dados para planilhas e sistemas institucionais, elevando o risco de inconsistências e extravios.

03

Falta de Transparência

Dificuldade para os alunos acompanharem o status de suas solicitações e falta de clareza no fluxo de validação.

Objetivos do Projeto (SHC)

Objetivo Geral: Desenvolver um sistema web para centralização, organização e apoio à gestão das atividades complementares, visando a digitalização, rastreabilidade e modernização dos processos acadêmicos.

✔

Digitalização de Formulários

Permitir cadastro e submissão com upload de comprovantes digitais (PDF).

✔

Rastreabilidade e Histórico

Implementar recursos para consulta e acompanhamento de status em tempo real.

✔

Fluxos de Validação

Criar mecanismos de análise para aprovação, ressalvas e reprovação.

✔

UX e Perfis de Acesso

Adequar interfaces e permissões aos diferentes atores envolvidos no fluxo.

Metodologia da Pesquisa

Pesquisa aplicada, exploratória e descritiva, com abordagem qualitativa. O objetivo foi a imersão profunda na realidade institucional para compreensão do processo e concepção da solução computacional.

1. Entrevistas Não Estruturadas

Realizadas com representantes dos setores envolvidos para identificar necessidades, regras institucionais e principais desafios enfrentados na execução das atividades.

2. Análise Documental

Estudo de regulamentos institucionais (ex: Resolução nº 004/2024 da FMP), formulários físicos e critérios de validação para embasar a regra de negócio do software.

3. Observação do Processo Atual

Acompanhamento in loco das etapas do fluxo institucional para mapear dificuldades operacionais, gargalos e pontos de retrabalho administrativo.

Modelagem: Perfis de Usuário

A arquitetura e os casos de uso do SHC foram divididos para atender as responsabilidades específicas de quatro atores principais:

Aluno

  • • Cadastro de atividades
  • • Envio de certificados
  • • Acompanhar status/histórico

Coordenador

  • • Avaliação e validação
  • • Adicionar ressalvas/feedback
  • • Consulta do histórico do curso

Sec. Acadêmica

  • • Histórico geral
  • • Gerenciamento de alunos
  • • Acompanhamento administrativo

Administrador

  • • Gerenciar configurações globais
  • • Criar categorias/cursos
  • • Gerenciamento de acessos

Engenharia e Modelagem

Requisitos Funcionais

  • Autenticação segura de usuários.
  • Submissão de atividades e anexação de documentos.
  • Validação e análise de solicitações pelos coordenadores.
  • Acompanhamento do histórico de registros.
  • Gerenciamento administrativo das informações.

Requisitos Não Funcionais

  • Aplicação de princípios de usabilidade e UX.
  • Segurança da informação e permissões em níveis.
  • Compatibilidade universal com ambientes web modernos.

Modelagem de Sistema

Utilização de padrões para estruturação do software antes da codificação:

UML

Diagramas de Casos de Uso e de Classe para descrever atores, funcionalidades e estrutura lógica.

MER/DER

Modelagem Entidade-Relacionamento para projetar a estrutura do banco de dados e integridade dos dados.

Tecnologias e Arquitetura

Stack de Desenvolvimento

Front-end / Prototipação

HTML, CSS, JS puros focados em usabilidade e design responsivo.

Back-end

PHP estruturado sobre o framework Laravel. Autenticação leve garantida pelo Laravel Sanctum.

Banco de Dados & Ferramentas

PostgreSQL (Relacional). Editor de código-fonte VS Code e controle de versão via GitHub.

Arquitetura MVC

Adoção do padrão Model-View-Controller no back-end isolando a lógica de negócio (validações, regras de carga horária) da camada de dados e de apresentação.

Segurança e Integração

Upload seguro apenas de formatos autorizados (PDF até 10MB) e estrutura preparada para integração via API com Sistemas Acadêmicos Externos.

Impacto: Análise Comparativa do Processo

Critério Processo Manual (Antes) Sistema SHC (Depois)
Cadastro / Envio Preenchimento manual físico / Entrega presencial. Formulário digital / Upload direto no sistema.
Retrabalho / Erros Elevado (transcrição de dados) / Alto risco de perda. Reduzido / Registro direto e centralizado com validação.
Acompanhamento Limitado e dependente da secretaria. Disponível em tempo real pelo aluno via Dashboard.
Tempo de Processamento Lento e dependente de etapas puramente manuais. Fluxo digital integrado, ágil e rastreável.

Validação Técnica e Usabilidade

Foi realizado um Teste Alfa baseado em observação de tarefas, validando aspectos funcionais, de navegação e aderência aos requisitos na versão homologada.

5 Usuários Testadores
2 Alunos 1 Coord. 1 Sec. 1 Admin

Resultados Positivos

Boa aceitação geral; facilidade de navegação; clareza na organização das telas; simplicidade no cadastro de atividades e facilidade no acompanhamento do histórico.

Oportunidades de Melhoria Aplicadas

Aprimoramento das mensagens de feedback durante ações; maior clareza visual para indicar cadastros prévios existentes, evitando dúvidas iniciais de interpretação.
